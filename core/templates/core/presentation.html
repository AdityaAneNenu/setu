2. Introduction

Project Overview, Problem Statement & Motivations

📋
Overview (PM-AJAY Scheme)
Over 112K SC-majority villages aim to become "Adarsh Grams". Our project tracks grievance redressal precisely for these areas to ensure real progress.
🏚️
Problem Statement
A road breaks. Someone writes it down. Complaints are easily lost, lack transparency, and sit in physical files forever. No way exists to trace closure automatically.
📡
Motivation: Network Deserts
Villages often get 2G at best. Any app needing internet stops working in the field, losing worker data due to dropped signals at the wrong moment.
🗣️
Motivation: Language Bias
Current portals serve English only. It is critical to build an inclusive system driven by voice and local languages for ground workers across India.
3. Background & Literature Review

Existing Solutions & Gaps Identified

Existing Solutions / Literature Survey
How grievance tracking is currently approached in rural areas.
1
Traditional Paper Registries at localized Post Offices.
2
Federal Web Portals (e.g., CPGRAMS) requiring typed inputs.
3
Ad-hoc Digital Communication via informal WhatsApp Groups.
4
Standalone feedback mobile apps (Syncs fail frequently).
Gaps Identified / Improvements
Why current systems fail the specific needs of Adarsh Grams.
!
No Offline Safety: Drops critical payload data if the connection is lost.
!
Language Hurdles: Manual typing limits reporting for non-English speakers.
!
Lack of True Proof: Anyone can easily falsify a "resolved" status remotely.
!
Manual Data Pipeline: Sluggish processing from paper trail to digital map.
4. Project Objectives

What we set out to achieve

🛡️
1. Resilient Async
Develop a 100% offline-first framework capturing payload on-device and automating background synchronizations resiliently when rural networks revive.
🤖
2. Automate Media with AI
Process rural grievance submissions autonomously utilizing Multimodal AI logic—allowing users to simply speak locally or capture visuals without typing.
🔒
3. Tamper-Proof Audit
Eradicate false resolutions using explicit geographical triangulation validation layered with biometric verification logic per individual worker.

5. Proposed Solution (Cont.)

Key Components & Features

📶
Offline Synchronization Protocol
Saves all data to local SQLite immediately. Uses polling loop listeners to blast queued payloads automatically on reconnect.
🎙️
Multilingual Voice Inference
AssemblyAI translates voice memos (7+ native languages) to english text -> Gemini maps categorization precisely.
📍
GPS Verification Block
Computes geographical distance mapping between the closure coordinate parameters vs. original report location markers.
🪪
Biometric Impersonation Defense
Selfies taken during closure are computed against registered face ID using neural CV algorithms to deter fraud.
📱
Paper Pipeline Integration
Post-office papers encode complaints inside generated QR matrices. Staff scan the printout to digitize offline traces instantly.
📊
Real-Time Data Aggregation
Automated admin dashboard fetching dynamic charts mapped directly continuously syncing off Firestore signals.

7. Summary

Findings, Limitations, & Future Directions

🔍
Findings
Core Deliverables
  • The offline-sync algorithm queues and protects state perfectly.
  • Voice accessibility eliminates manual literacy friction completely.
  • Multilayer checks (GPS/Face) vastly deters premature gap statuses.
⚠️
Limitations
Current Hurdles
  • Susceptible to device timestamp meddling natively.
  • Acoustic processing necessitates cloud propagation eventually resulting in pipeline waits.
  • On-device payload size can escalate during sustained drop zones.
🚀
Future Directions
The Road Ahead
  • Implement edge-based On-Device LLMs to skip network wait limits entirely.
  • Bot integration capabilities extending to WhatsApp and regional mediums natively.
  • Platform abstraction handling grievances across generalized domains.
